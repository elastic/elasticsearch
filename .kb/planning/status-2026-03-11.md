# ES|QL External Data Sources — Status March 11, 2026

---

## Slide 1: What We Are Building

**Headline**: Query data in cloud storage directly from ES|QL — no ingestion required

Observability and log analytics users have data that lives outside Elasticsearch — archived logs in S3 buckets, exported metrics in GCS, compliance data in Azure Blob. Today, querying that data means ingesting it first. External Data Sources removes that barrier — query cold and warm data in place, alongside hot data in Elasticsearch.

```
EXTERNAL "s3://my-bucket/logs/*.parquet"
| WHERE @timestamp > NOW() - 1 day
| STATS count(*) BY status_code
```

**The product**: A user points ES|QL at files in cloud storage and queries them with the full power of the ES|QL language — the same WHERE, STATS, SORT, EVAL, JOIN, and ENRICH they already know. No new query language. No separate compute engine. No data movement.

**What it supports**:
- **Any major cloud store** — S3, GCS, Azure Blob, HTTP(S)
- **Any common file format** — Parquet, CSV, NDJSON, ORC — including compressed (GZIP, ZSTD, BZIP2)
- **Named datasources** — register a datasource once with `CRUD`, store credentials securely, query by name
- **Schema handling** — automatic type inference, cached metadata for dashboard performance, union-by-name for files with evolving schemas
- **Production-safe** — external queries can't starve search workloads, memory is bounded, execution is distributed across the cluster

**Target**: MVP GA in ES 9.5 (June 2026). Tech Preview ~May 2026.

**Post-GA roadmap**: Iceberg catalog, Arrow-native Parquet reads, Flight/gRPC connectors, SQL database federation, schema discovery for Kibana autocomplete.

## Slide 2: Where We Are — What Works Today

**Headline**: From zero to end-to-end queries in 4 weeks. ~50 PRs, ~140K lines, 13 plugins.

**You can today** run this query and get correct results:
```
EXTERNAL "s3://my-bucket/logs-2026-03/*.parquet"
| WHERE status_code >= 500 AND @timestamp > NOW() - 1 hour
| STATS error_count = count(*), avg_latency = avg(duration_ms) BY service_name
| SORT error_count DESC
| LIMIT 20
```

**What works today:**
- **Read any format**: Parquet, CSV, NDJSON, ORC
- **From any cloud store**: S3, GCS, Azure Blob, HTTP(S) — with glob patterns (`*.parquet`) and compressed files (GZIP, ZSTD, BZIP2)
- **Full ES|QL language**: WHERE, EVAL, STATS, SORT, LIMIT, KEEP, DROP, RENAME, DISSECT, GROK, MV_EXPAND, ENRICH, LOOKUP JOIN, INLINESTATS, FORK — 30+ commands and all scalar functions
- **Distributed automatically**: Queries split across data nodes based on file count and size
- **Smart execution**: Only reads columns the query needs. Stops reading once LIMIT is satisfied. Aggregations run locally and merge globally.
- **Arrow-native compute**: Bridge between Arrow memory and ES|QL engine (merging this week)

**Project management**: 34 tracking issues with estimates, ownership, and dependencies. Swim lanes defined. Milestones established. Velocity measured against actuals.

## Slide 3: Where We Are — What's Left to Build

**Headline**: Infrastructure done. Remaining work: correct every type, secure it, prove it's fast, make it a product.

| What's left | Why it matters | Status |
|-------------|---------------|--------|
| **Type correctness** (19 issues across 4 formats) | Parquet DECIMAL, timestamps, INT96; NDJSON nested objects, nulls; CSV headers, datetime; ORC DECIMAL, LIST — users need correct data | Characterized with reproduction tests. 10.3w. |
| **Secrets** | Users need to store S3/GCS/Azure credentials securely, not inline in queries | Not started. 3.0w. |
| **CRUD API** | Named datasources — `EXTERNAL "my-logs"` instead of `EXTERNAL "s3://..."` | Not started. 2.0w. |
| **EXTERNAL syntax ungating** | Currently behind a dev flag — needs to work in production builds | Starting this month. 0.5w. |
| **Search workload protection** | External queries must not degrade Elasticsearch search performance | Not started. 1.0w. |
| **Memory safety** | Circuit breaker on source buffers — large files must not OOM the cluster | Not started. 2.0w. |
| **Rally + JMH** | Performance regression detection and format reader throughput benchmarks | Not started. 3.0w. |
| **Distributed hardening** | Fault tolerance under node failures during external source queries | Not started. 2.0w. |
| **Metadata cache** | Cache inferred schemas — don't re-read file headers on every query | Not started. 3.0w. |
| **Union-by-name** | Handle schema drift across files (missing columns, type differences) | Not started. 3.0w. |
| **Parquet-MR improvements + CB** | Faster reads and memory-safe Parquet execution | Not started. 6.0w. |

# Refactoring Plan 5: Unix Domain Socket IPC for LaunchDescriptor

## Goal

Replace the temp file IPC between the launcher and preparer with a Unix domain socket.
The launcher creates a `ServerSocketChannel` on a UDS path, passes the path to the preparer
via a `--launch-socket` command-line argument, and uses a `Selector` to wait for either the
descriptor or process exit. This eliminates the temp file, the `setupTempDir()` function from
the launcher, and the `ES_TMPDIR` injection into the preparer's environment.

## Current vs New Flow

**Current:** launcher creates temp dir, spawns preparer, waits for exit, reads descriptor from
temp file, deletes file.

**New:** launcher creates a UDS, spawns preparer with `--launch-socket <path>`, reads descriptor
from accepted socket connection while waiting for process exit via a `Selector`.

### Current process relationships

```
bin/elasticsearch (bash)
  └── exec: Launcher (server-launcher, native or JVM)
        ├── setupTempDir()                            ← creates ES_TMPDIR
        ├── fork/exec: Preparer (server-cli JVM)
        │     ├── Writes launch-descriptor.bin to ES_TMPDIR
        │     └── Exits
        ├── Reads launch-descriptor.bin from ES_TMPDIR
        ├── Deletes launch-descriptor.bin
        └── fork/exec: Elasticsearch Server JVM
```

### New process relationships

```
bin/elasticsearch (bash)
  └── exec: Launcher (server-launcher, native or JVM)
        ├── Creates ServerSocketChannel on java.io.tmpdir/es<pid>.sock
        ├── Starts selector thread (OP_ACCEPT + cancel pipe)
        ├── fork/exec: Preparer (server-cli JVM, --launch-socket <path>)
        │     ├── Connects to UDS, writes descriptor, closes
        │     └── Exits
        ├── Selector thread reads descriptor from accepted connection
        ├── Joins selector thread, gets descriptor
        ├── Cleans up socket file
        └── fork/exec: Elasticsearch Server JVM
```

The launcher no longer creates or manages temp directories. The socket is placed in the
system's existing temp directory (`java.io.tmpdir`) with a short PID-based name, staying
well within the ~104-byte Unix domain socket path length limit.

## Why Unix Domain Sockets Over Named Pipes (FIFOs)

REFACTORING_PLAN_4 dismissed named pipes, citing the lack of a JDK API for `mkfifo`, the
absence of POSIX FIFOs on Windows, and the need for careful open ordering to avoid deadlock.
While Windows does have named pipes (`\\.\pipe\`), the creation API differs from `mkfifo`.

We explored FIFOs again now that the launcher controls both sides. The fundamental problem is
that Java's `FileChannel` is not a `SelectableChannel`, so there is no way to multiplex a
FIFO read with a cancellation signal using a `Selector`. The reader thread would block on
`open()` with no clean way to cancel it if the preparer exits without writing (e.g., error
cases). Workarounds exist (open the write side to deliver EOF) but are untidy.

Unix domain sockets solve all of these problems:

- **JDK-native**: `UnixDomainSocketAddress` (JDK 16+) requires no external tools or JNI.
- **Cross-platform**: Works on Linux, macOS, and Windows 10+ — no platform-specific fallback.
- **Selectable**: `ServerSocketChannel` and `SocketChannel` work with `Selector`, enabling
  clean multiplexing of the accept and a cancellation signal.
- **No deadlock concerns**: The `Selector` wakes up on either event; no thread blocks
  indefinitely.

## Detailed Changes

### 1. Update `LaunchDescriptor`

**File:** `libs/server-launcher-common/src/main/java/.../LaunchDescriptor.java`

- Remove `DESCRIPTOR_FILENAME` constant (the socket path is generated by the launcher and
  passed via `--launch-socket`, so no shared constant is needed).
- Remove `writeTo(Path)` and `readFrom(Path)` file-based convenience methods.
- Keep `writeTo(DataOutputStream)` and `readFrom(DataInputStream)` — these are stream-based
  and work identically over a socket stream.

### 2. Restructure `ServerLauncher.main()`

**File:** `distribution/tools/server-launcher/src/main/java/.../ServerLauncher.java`

Remove `setupTempDir()` entirely. New main flow:

```
main(args)
  if args[0] == "--dump" and args[1] is a path:
    read descriptor from path, dump, exit           // existing diagnostic mode

  socketPath = java.io.tmpdir / "es<pid>.sock"
  serverSocket = ServerSocketChannel.open(UNIX)
  serverSocket.bind(UnixDomainSocketAddress.of(socketPath))
  cancelPipe = Pipe.open()

  selectorThread = start thread:
    selector = Selector.open()
    serverSocket.register(selector, OP_ACCEPT)
    cancelPipe.source().register(selector, OP_READ)
    selector.select()
    if OP_ACCEPT:
      client = serverSocket.accept()
      descriptor = LaunchDescriptor.readFrom(new DataInputStream(...))
      return descriptor
    if OP_READ (cancel):
      return null

  preparerExit = runPreparer(args, socketPath)   // passes --launch-socket <path>

  if selectorThread.isAlive():
    cancelPipe.sink().write(ByteBuffer.allocate(1))  // wake selector
  selectorThread.join()
  descriptor = selectorThread.result

  cleanup: close serverSocket, delete socketPath

  if preparerExit != 0: exit with that code
  if descriptor == null: return                  // e.g. --version was used

  registerShutdownHook()                         // same as before
  startServer()                                  // same as before
  waitFor() or detach()                          // same as before
```

Key changes to `runPreparer()`:

- Append `--launch-socket <socketPath>` to the preparer command line.
- No longer set `ES_TMPDIR` in the preparer's environment. The preparer handles its own
  temp dir independently via `ServerProcessUtils.setupTempDir()`.

### 3. Update `ServerCli`

**File:** `distribution/tools/server-cli/src/main/java/.../ServerCli.java`

- Add `--launch-socket` option to the jopt-simple parser. This option accepts a path argument
  and is hidden from help output (it is an internal coordination parameter, not a user-facing
  flag).
- In `prepareLaunch()`: read the socket path from the option, connect via
  `SocketChannel.open(UnixDomainSocketAddress.of(path))`, write the descriptor using
  `LaunchDescriptor.writeTo(DataOutputStream)` wrapping the socket's output stream
  (`Channels.newOutputStream(socketChannel)`), close the channel.
- Remove the file-based `descriptor.writeTo(descriptorPath)` call.
- `prepareLaunch()` still calls `ServerProcessUtils.setupTempDir()` for JVM options and
  `LIBFFI_TMPDIR` — that is unchanged.

### 4. No changes needed elsewhere

- **`ErrorPumpThread`**, **`ServerProcess`** — unchanged.
- **`WindowsServiceDaemon`** — bypasses the launcher entirely (directly spawns the server
  in-process); unaffected.
- **`bin/elasticsearch`**, **`bin/elasticsearch.bat`** — unchanged (launcher handles
  everything).
- **Native image build** — `ServerSocketChannel`, `UnixDomainSocketAddress`, `Selector` are
  all supported by GraalVM native-image.

## Key Design Decisions

- **`--launch-socket` command-line argument**: The socket path is a launcher-internal
  coordination detail, not a server configuration property. Passing it as a CLI argument
  (rather than a `-Des.*` system property) keeps it distinct from properties that configure
  the Elasticsearch environment. The option is hidden from help output.

- **Socket path in `java.io.tmpdir`**: The socket is extremely short-lived (exists only during
  startup, used once, deleted). Placing it in the system's existing temp directory with a
  short PID-based name (`es<pid>.sock`) avoids creating or managing any directories and stays
  well within the ~104-byte path length limit on all platforms.

- **No `setupTempDir()` in launcher**: The launcher's `setupTempDir()` existed solely to
  create a shared directory for the descriptor file. With UDS, the launcher just places a
  socket file in an already-existing directory. The preparer independently manages its own
  temp dir via `ServerProcessUtils.setupTempDir()`.

- **`Selector` for multiplexing**: Cleanly waits on either a descriptor connection (OP_ACCEPT)
  or a cancellation signal (OP_READ on the cancel pipe). No polling, no thread interrupt
  hacks, no blocked `open()` calls.

- **Cancellation via `java.nio.channels.Pipe`**: When `waitFor()` returns and the selector
  thread is still alive (preparer exited without connecting), the main thread writes a byte
  to the pipe's sink channel. The selector wakes up, the thread returns null, and cleanup
  proceeds.

- **Stream-based serialization unchanged**: `LaunchDescriptor.writeTo(DataOutputStream)` and
  `readFrom(DataInputStream)` work identically over a socket stream as they did over a file
  stream. The preparer does not need to know or care that the other end is a socket.

- **`--dump` mode preserved**: `server-launcher --dump <descriptor-path>` continues to work
  for inspecting descriptor files that were saved separately (e.g., for debugging). This
  code path uses its own `readFrom(Path)` or can be adapted to read from any input stream.
